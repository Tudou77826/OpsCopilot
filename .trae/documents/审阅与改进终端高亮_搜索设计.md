## 审阅结论（针对 terminal-highlight-search-design.md）
- 方向正确：用 xterm-addon-search 实现 Ctrl+F 搜索；用 xterm Decoration 实现高亮规则；并考虑增量/上限/可见区优化，符合 DevOps 场景。
- 但当前文档仍偏“方案罗列”，缺少与现有代码的对齐与可落地约束，导致不是最优解：
  - 依赖与仓库现状不一致（当前未安装 xterm-addon-search）。
  - 与现有 Terminal 键盘拦截/补全 overlay 存在冲突点，未定义优先级。
  - “增量高亮”缺少可靠触发点（输出写入路径在现有实现里不是 event-driven）。
  - Regex 安全策略不可仅依赖 AI（JS regex 无超时），需本地硬限制。
  - 配置落盘方案需对齐现有 config.json + prompts/quick_commands 拆分保存模式。

## 我将先做的文档修订（不写代码）
### 1) 增补“现状对齐”章节（必须落地）
- 补充当前仓库事实：React 18 + xterm 5.3 + xterm-addon-fit 已存在；SearchAddon 未安装；Terminal 组件已接管 Ctrl+C/Ctrl+V/Tab/方向键与补全 overlay。
- 明确 Ctrl+F 的键盘事件链路：Terminal key handler → 是否拦截 → SearchPanel 状态机。

### 2) 将功能拆成“可验证的 MVP 行为规格”
把模糊描述改成可以写测试的规格（Given/When/Then），包括：
- 搜索面板打开/关闭（Ctrl+F、Esc、点击×）
- 导航规则（Enter/Shift+Enter/F3/Shift+F3）
- 选项（大小写/正则）切换对结果的影响
- 搜索高亮与规则高亮的层级规则（谁覆盖谁、是否叠加）

### 3) 给出“高亮引擎”的明确数据流与降级策略
把“监听数据写入事件”改成工程可实现的策略二选一并定稿：
- 推荐默认：write() 后节流触发“可见区±N 行扫描”，不依赖 xterm 内部回调。
- 备选：onRender 事件触发扫描（更贴近渲染但更复杂）。
同时定义硬约束（写入文档）：
- 每条规则 pattern 长度上限、每行最大匹配次数、总装饰器上限、每轮扫描时间预算（超预算则跳过并提示“已降级”）。

### 4) 重新定位“AI 正则校验”的角色
- AI 只做“建议/风险提示”，不能作为唯一安全机制。
- 文档补充本地静态检查清单（嵌套量词、回溯热点模式等）与保存时的强校验策略。
- 明确无 LLM 配置时的行为：按钮禁用或提示，不阻断功能。

### 5) 配置落盘方案与 UI 入口定稿
- 落盘：对齐 prompts.json/quick_commands.json 模式，设计 highlight_rules.json 的读写与默认生成。
- UI：不依赖拖拽库，首版采用“上下箭头调序”；支持导入模板、测试文本预览。
- 明确“全局禁用高亮”的总开关（紧急止损）。
