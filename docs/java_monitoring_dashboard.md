# Java 监控面板（无新增依赖）需求分析 & 功能设计

## 1. 背景与目标

### 1.1 背景
当前服务器主要运行 Java 程序，排障与容量评估常依赖手工登录机器、执行 `top/ps/jstat/jcmd` 等命令，信息分散且难以沉淀为统一视图。

### 1.2 目标
在 OpsCopilot 内提供一个“单独入口”的 Java 监看页面，面向已建立的 SSH 会话，基于 **JDK8/JDK21 自带工具** 与 **Linux 指令**快速汇总 JVM/进程/主机关键状态，实现：
- 1 分钟内判断：主机资源问题 / JVM(GC/内存/线程) 问题 / 应用异常
- 低侵入：不要求应用改造、不新增 Agent、不引入新服务与依赖
- 可扩展：后续可逐步增加更深的诊断卡片与联动

### 1.3 非目标（首期不做）
- 自研时序存储、采集链路、告警平台、分布式追踪
- 复杂的可视化拖拽编辑器与全量 APM 能力

## 2. 约束与原则

### 2.1 约束
1. 仅依托 **JDK8 或 JDK21** 自带内容（如 `jcmd/jstat/jps/jstack/jmap`），不引入新的第三方依赖或服务端组件。
2. 仅使用 **Linux 常见命令**（如 `ps`, `cat`, `grep`, `awk`, `sed`, `lsof/ss` 等；实际支持以目标系统是否存在为准）。
3. 监控采集以 **当前 SSH 会话所在机器**为准（通过已连接的会话执行命令获取信息）。

### 2.2 设计原则
- “可用优先”：先提供可靠的关键指标与文本视图，再逐步结构化与图表化
- “可降级”：JDK 工具缺失或权限不足时，给出可解释的提示并使用替代 Linux 视图
- “低风险”：默认只执行只读命令；高开销命令需用户显式触发

## 3. 角色与使用场景

### 3.1 角色
- 运维：快速判断资源水位、定位异常实例、辅助值班
- 开发：快速查看 GC/内存/线程状态，判断是否需要重启/扩容/代码修复

### 3.2 核心场景
- CPU 飙高：判断是 Java 进程占用还是系统层瓶颈
- GC 频繁：查看堆使用、GC 统计、回收器信息
- OOM 风险：堆接近上限、线程数/FD 逼近限制
- 进程异常：Java 进程重启、卡死、线程阻塞/死锁（可选）

## 4. 功能需求（FR）

### FR-1 独立入口与会话绑定
- 在软件中提供单独入口进入“监控”页面
- 默认以“当前激活的终端会话”作为监控目标（可切换到其它已连接会话）

### FR-2 Java 进程发现与选择
- 列出当前机器上的 Java 进程列表：PID、启动命令（截断）、所属用户、运行时长等
- 支持选择一个 PID 进入详情监看

### FR-3 快照式监控（MVP）
对选定 PID 生成一份可刷新的“快照”，包含：
- 进程信息：CPU/MEM、运行时长、线程数、FD 数
- JVM 信息（尽力）：JDK 版本、启动参数（精简）、堆与 GC 摘要
- 主机信息（可选）：Load、内存、水位（用于判断是否系统层问题）

### FR-4 安全与权限提示
- 命令执行失败时，展示失败原因（工具不存在/权限不足/进程不存在）
- 对可能泄露敏感信息的内容（如 `VM.system_properties`）默认不采集或仅显示白名单字段

### FR-5 可选“深度诊断”（后续增强）
用户点击后执行，且提示可能带来开销：
- `jstack` 线程 dump（截断/下载）
- `jcmd Thread.print` / `jcmd GC.class_histogram` 等
- “高 CPU 线程定位”辅助命令组合（`top -H -p PID` 等）

## 5. 非功能需求（NFR）
- 性能：单次快照默认应在 2~5 秒内完成（取决于网络/远端负载）
- 可靠性：任一子命令失败不影响整体页面渲染（分块降级）
- 安全：不记录明文密码；敏感输出不默认落盘

## 6. 技术方案（无新增依赖）

### 6.1 数据获取方式
通过 OpsCopilot 已建立的 SSH 会话，在后端发起“非交互命令执行”，读取 stdout 并返回前端渲染。

### 6.2 命令集合（MVP 默认执行）
以下命令均应以“存在即用”为原则（先检测 `command -v` 或执行失败降级）：

**进程基础信息**
- `ps -p <pid> -o pid=,ppid=,user=,%cpu=,%mem=,etime=,cmd=`
- `cat /proc/<pid>/status`（解析 Threads/VmRSS 等）
- `ls /proc/<pid>/fd 2>/dev/null | wc -l`

**主机基础信息（可选）**
- `uptime`
- `cat /proc/meminfo | head -n 20`

**JDK 工具（尽力）**
- `command -v jcmd && jcmd <pid> VM.version`
- `command -v jcmd && jcmd <pid> GC.heap_info`
- `command -v jstat && jstat -gcutil <pid> 1000 1`

### 6.3 安全策略
- 默认不执行 `jcmd <pid> VM.system_properties`（可能包含路径/参数/密钥痕迹）
- 所有命令均为只读；深度诊断需显式点击触发
- 对输出做长度上限限制（防止过大导致 UI 卡顿）

## 7. 页面与交互设计

### 7.1 导航入口
- 右侧导航栏增加一个“监控”图标入口，打开侧边栏的“监控”标签页。

### 7.2 监控页结构（侧边栏面板）
1. 顶部：会话选择（默认当前激活会话）、刷新按钮、自动刷新开关（后续）
2. 中部：Java 进程列表（点击选择 PID）
3. 底部：选中 PID 的快照卡片（分块显示：进程、JVM、主机）

### 7.3 错误与降级展示
- 未选择会话：提示“请先建立连接并选中一个终端”
- 未发现 Java 进程：提示“当前机器未发现 java 进程或权限不足”
- JDK 工具缺失：提示“远端未安装 JDK 工具（jcmd/jstat），仅展示 Linux 进程信息”

## 8. 接口设计（前后端）

### 8.1 ListJavaProcesses
输入：`sessionID`
输出：JSON（或 `Error:` 字符串）

字段建议：
- `pid`（number）
- `user`（string）
- `etime`（string）
- `cmd`（string，截断）

### 8.2 GetJavaMonitorSnapshot
输入：`sessionID`, `pid`
输出：JSON（或 `Error:` 字符串），分块返回：
- `process`：进程卡片
- `host`：主机卡片（可选）
- `jvm`：JVM 卡片（尽力）
- `tools`：工具可用性与命令执行错误列表

## 9. 兼容性说明
- JDK8：`jcmd` 在部分发行版可能不随 JRE 提供；需安装 JDK 或配置 PATH
- JDK21：大多数工具可用，但容器化场景可能权限受限
- 最低权限：若无读取 `/proc/<pid>` 权限，则展示可用部分并提示

## 10. 后续演进方向
- 增加“诊断卡片”：高 CPU 线程定位、死锁检测、GC 原因提示
- 增加“命令复用”：将快照使用的命令生成可复制的诊断脚本
- 增加“导出”：将一次快照导出为 Markdown 复盘材料（可接入现有 docs 体系）
