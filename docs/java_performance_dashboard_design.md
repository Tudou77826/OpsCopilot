# Java 性能看板（Java Performance Dashboard）功能设计（含 UI/UX）

> 目的：先把“这个功能到底要长什么样、解决什么决策问题”讲清楚，再进入实现迭代。本文只描述设计，不涉及代码与实现细节。

## 0. 已确认的产品决策（讨论结果）

- 主要用户：开发人员（但核心诉求是定位/监控性能指标，帮助排障决策）
- 默认时间范围：近 15 分钟（可切换）
- 默认模式：单机单 PID 深挖（暂不做多机对比）
- 工具缺失策略：隐藏 GC 区块保持简洁（不刷屏提示；仅在“工具可用性”区域提示增强路径）

## 1. 背景与目标

### 1.1 背景
- 命令行（`top/ps/jstat/jcmd` 等）可以看到信息，但信息碎片化、对比困难、无法沉淀为统一视图。
- 看板的价值不应是“把命令输出搬到 UI”，而应是：用更少、更聚焦的信息回答更关键的问题，并提供下一步动作入口。

### 1.2 核心目标（以决策为中心）
用户打开看板的 30 秒内，应能回答：
1. 这是**机器资源瓶颈**还是**JVM/应用自身问题**？
2. 这是**持续恶化**还是**短暂抖动**？
3. 更可能的方向：**CPU 热点 / GC 压力 / 内存泄漏趋势 / 线程阻塞 / I/O 等待**？
4. 下一步该点哪里（钻取）才能更快收敛？

### 1.3 非目标（明确不做）
- 不做全量 APM（链路追踪、调用树、慢 SQL 自动归因）
- 不做自研时序存储/告警平台
- 不做“所有指标都展示”的堆砌式大屏

## 2. 约束与数据来源（决定看板上限）

### 2.1 约束
- 仅依赖 **JDK8/JDK21 自带工具**与**Linux 指令**（不引入新依赖、agent、后端服务）。
- 数据来自“已建立的 SSH 会话”，以目标机器为准。

### 2.2 数据能力分层（必须支持降级）
为避免“一堆 Error=废品”，定义分层并在 UI 上可视化：
- **L1（必有）**：`ps`、`/proc`、`uptime`、`/proc/meminfo`  
  - 能做：CPU/Mem/Threads/FD/Load 趋势、基本健康判断
- **L2（增强）**：`jstat`、`jcmd`  
  - 能做：OldGen 使用趋势、GC 压力、JVM 信息摘要
- **L3（手动触发）**：`jstack`、`jcmd Thread.print`、`jcmd GC.class_histogram` 等  
  - 能做：线程 dump、类直方图对比（注意开销/权限）

**设计原则：**
- 看板在 **L1 也必须“有用”**；L2/L3 是加分项，通过“工具可用性徽标 + 按钮”逐步增强。
- 权限：运行看板的用户应与 Java 进程用户一致（或具备足够权限），否则 `jstat/jstack/jcmd` 可能无法 attach。
- 性能：`jstat` 可高频（例如 1s/3s）；`jstack` 较重，必须做成“手动触发”或低频/条件触发（如 CPU 持续高于阈值才采样一次）。

## 3. 用户与使用场景

### 3.1 用户
- 开发：需要快速定位性能问题方向（CPU/GC/内存/线程/句柄/网络），并决定下一步动作（采样/抓栈/输出证据）

### 3.2 典型场景 → 第一屏必须给出的答案
1. **CPU 飙高**：进程 CPU% 趋势 + Load +（可选）线程数变化 + 最近 GC 开销  
2. **频繁 Full GC / 停顿**：Old 使用率趋势 + FGC 次数/间隔 + GC 开销趋势  
3. **疑似内存泄漏**：Old 使用率单调上升 + 进程内存趋势（RSS 或 %）  
4. **应用卡死/线程阻塞**：线程数变化 + 一键线程 dump（手动触发，提示风险）

## 4. 信息架构（IA）：三层结构

### 4.1 层 0：入口与上下文（顶部固定区）
顶部固定区（sticky）必须永远可见，包含：
- 目标会话（机器/连接）
- 目标 Java 进程（PID + 命令摘要）
- 时间范围（默认近 15m，可选近 5m / 1h）与刷新频率（3s/5s/10s/手动）
- 工具可用性徽标：例如 `jstat ✓ / jcmd ✓ / jstack ×`（决定显示哪些卡片/图表）

### 4.2 层 1：总览（必须强、禁止凑数）
总览区必须回答三个问题（同事建议非常关键，吸纳为总览组织原则）：
1) **系统活得好吗？**（资源饱和度）  
2) **Java 代码卡吗？**（GC 与线程）  
3) **有没有泄露风险？**（句柄与堆/非堆趋势）  

#### 4.2.1 核心生命体征（置顶：一眼定生死）
这块是“置顶区域”，只展示对定位最直接的结果，且要尽量区分“业务负载 vs GC负载”。

**CPU（区分业务 vs GC）**
- 进程 CPU% 趋势（必有）
- 若 `jstat` 可用：展示 **GC 时间占比趋势**（用 `GCT/FGCT/YGCT` 的变化率推算）  
- 若系统侧能取到：用户态 vs 内核态占比（Sys 高常见于 I/O 或上下文切换）

**内存画像（RSS vs Heap vs Non-Heap）**
- 物理内存 RSS（必有，来自 `/proc/<pid>/status` 的 `VmRSS`）趋势
- 若 `jstat` 可用：Heap 使用趋势（可拆分 Eden/Survivor/Old；但默认不展开细分）
- 若 `jcmd` 可用：Non-Heap/Metaspace 概览（可选，建议作为 drilldown 内容）

**句柄健康度**
- FD 使用量 / 上限（必有：`/proc/<pid>/fd` 数量；上限来自 `ulimit -n` 或 `/proc/<pid>/limits`）

#### 4.2.2 总览趋势图（默认展示集合）
总览趋势图遵循“少而精”，且必须与动作挂钩。

**默认展示（L1 可用）**
1) CPU%（进程）趋势  
2) RSS（进程物理内存）趋势  
3) Threads（线程总数）趋势  
4) FD（句柄数）趋势 + 上限线（如果能取到）  

**仅在 `jstat` 可用时显示（隐藏 GC 区块保持简洁）**
5) OldGen 使用%（O）趋势  
6) GC interval cost：每个刷新间隔的 ΔGCT 换算 ms（提示“是否在 GC 上耗时”）  
7) FGC 计数/速率（红线指标，低频显示为徽章或小图）  

**为什么默认只强调 Old%（而不是把 Eden/S0/S1 全部铺开）**
- Old% 是“长期趋势”最有价值的信号：  
  - Old 在 GC 后水位持续抬升，往往是内存压力/泄漏风险最直观的早期指标；也更适合 15 分钟窗口的深挖。
- Eden/S0/S1 并非不重要，而是更偏“短周期分配压力”与“抖动特征”：  
  - Eden 高通常代表分配快、Young GC 频繁；但它波动很大，放在总览容易造成噪声。更适合放在 **GC Drilldown 的分代细分视图**里。
- Metaspace/Non-Heap 也很重要（类加载/代理/脚本导致的泄漏风险），但在“无新增依赖 + 工具不一定齐全”的约束下：  
  - 建议作为 **drilldown（可选显示）**，避免总览被“工具缺失/权限不足”污染。

#### 4.2.3 结论条（规则化、可解释）
结论条用于把“图”翻译成一句话，减少用户脑内计算。必须可解释、可复现。

示例规则（可讨论迭代）：
- `CPU > 85% 持续 3 个采样点 且 ΔGCT 很小` → `疑似业务 CPU 热点/死循环，优先定位 Top CPU Threads`
- `OldGen 使用上升 + FGC 增长 或 ΔGCT 上升` → `疑似 GC 压力/内存泄漏趋势，优先查看 Old 趋势与（手动）线程/类直方图`
- `FD 接近上限 或 TIME_WAIT 激增` → `疑似连接未释放/句柄泄漏风险，优先看连接状态与 FD 走势`

**关键告警徽章（轻量，最多 3~5 个）**
- CPU 高、RSS 高、Threads 异常、FD 接近上限、（jstat 可用时）FGC 增长/GC 开销高

### 4.3 层 2：钻取（第二屏/展开区）
按“问题方向”组织，而不是按“命令来源”组织：
- **CPU Drilldown**
  - 展示 CPU% 与 load 的更细粒度解释
  - 生成建议命令（或手动触发采样）用于定位高 CPU 线程/热点
  - 建议重点吸纳同事方案中的“经典组合拳”：Top 线程（OS TID）→ 十六进制 → 关联到 Java 线程栈（仅手动触发）
- **GC Drilldown**
  - `jstat -gcutil` 字段表格化（含解释：O/Y/M/CCS/YGC/FGC/GCT）
  - 关注“速率”：FGC/min、Old 上升斜率、GC 开销随时间变化
- **线程 Drilldown**
  - 线程数趋势（L1 可得）
  - 线程状态计数（RUNNABLE/BLOCKED/WAITING/TIMED_WAITING）：
    - 优先策略：手动触发一次 `jstack` 并解析（不要高频）
  - 手动触发线程 dump（L3），并做输出截断/下载
- **进程/系统 Drilldown**
  - FD 数趋势、meminfo 关键行结构化提取、可选网络/磁盘指标入口
  - 不包含网络方向内容（本部署环境下价值较低，明确不纳入范围）

### 4.4 层 3：证据与复盘（默认隐藏）
- 原始命令输出“只作为证据”，以可折叠方式呈现
- 支持“一键复制/导出诊断快照”（后续）

## 5. UI/UX 规范（直观 + 模块化）

### 5.1 布局（侧边栏形态）
- 顶部固定区（sticky）：上下文选择器
- 内容滚动区（唯一滚动容器）：总览卡片 → 结论卡片 → drilldown → 证据
- 每个信息单元 = 一个独立 Card 模块：
  - 标题 + 右上角动作（刷新/导出/触发采样）
  - 图表区 + 图例/阈值线 + 当前值/变化率

### 5.1.1 推荐卡片编排（单机单 PID 深挖）
从上到下建议固定顺序，减少用户寻找成本：
1) 核心生命体征（CPU/RSS/Threads/FD + 关键徽章）
2) 趋势图（CPU、RSS、Threads、FD）
3) 结论条（规则解释 + 推荐下一步动作按钮）
4) Drilldown：
   - Top CPU Threads（默认可见模块，但需要手动触发采样；展示 Top N + 状态 + 线程名/栈片段）
   - GC（仅 jstat 可用时显示）
   - 线程状态（手动触发；或 CPU 持续超阈后自动采样一次并提示）
   - FD（可选深挖）
5) 证据区（原始输出折叠）

### 5.2 图表规范（无依赖也要“像监控”）
每张图最少应包含：
- 折线 +（可选）轻量面积填充
- 单位与当前值（右下角）
- 阈值线（例如 80%、90%）+ 超阈变色
- Hover（后续增强）：显示点值与时间

建议增加：
- 上限/红线：FD 上限线、Old 使用红线（jstat 可用时）、CPU 90% 红线
- 采样一致性提示：本页数据是“采样快照”，不是完整时序存储（避免误解）

### 5.3 指标准入规则（避免凑数）
只有满足以下至少一条的指标才能进入总览：
- 能直接触发动作：扩容/重启/抓 dump/继续钻取
- 能区分方向：CPU 热点 vs GC 压力 vs I/O 等待（在约束下尽力）
- 能做趋势判断：持续、抖动、阶跃（变更影响）

### 5.4 空态与降级体验
- 未连接会话：引导用户建立连接
- 未选择 PID：引导从 Java 进程列表选择
- `jstat/jcmd` 缺失：隐藏 GC/JVM 深度区块保持简洁；顶部徽标提示“安装 JDK 工具可增强”
- 权限不足：证据区给出可复制的解决建议（例如需同用户/需 root）

## 6. 功能清单（按版本拆分）

### MVP（最小可用，必须“能用”）
- 会话选择 + Java 进程选择（这里需要注意，一个机器上的java进程可能存在上百个，用户需要能够快速筛选出自己关心的进程）
- 自动刷新（默认 5s）+ 手动刷新
- 趋势图（L1）：CPU%、RSS、Threads、FD（含上限线，若可得）
- （可选，jstat 可用时显示）Old%、GC interval cost、FGC 计数/速率
- 规则结论条（可解释）
- 原始证据折叠区（可复制）

### V1（提升排障效率）
- GC 表格结构化（字段解释）+ GC 吞吐量估算（可解释公式）
- Top N 最忙线程定位（CPU 飙高救命功能，手动触发）
- 一键抓线程 dump（手动触发 + 风险提示 + 输出截断/下载）
- 导出“诊断快照”Markdown

## 7. 交互流程（关键路径）

### 流程 A：值班快速判断
1) 打开看板 → 选择会话 → 选择 PID  
2) 看 4 图 + 结论条  
3) 若结论指向 GC：展开 GC Drilldown → 必要时抓 dump  
4) 导出快照发群/工单

### 流程 B：开发深入排查
1) 选 PID → 看 CPU/GC/Old 趋势  
2) CPU 高且 GC 低：走 CPU Drilldown（建议命令/采样）  
3) Old 上升：抓 class histogram（后续）或至少线程 dump

## 8. 实验开关与灰度策略
- 看板作为实验功能默认关闭
- 开启后出现入口与页面
- 页面顶部标识“Experimental”，提供反馈入口（后续）
